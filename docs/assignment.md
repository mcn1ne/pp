## ① 문제 정의

### 해결하려는 문제

슈퍼센트는 게임 홍보를 위해 YouTube 파트너 크리에이터를 모집하고 있다. 현재 마케팅 팀은 크리에이터 지원자를 **수작업으로 평가**하고 있으며, 그 과정에서 다음과 같은 문제가 발생한다.

- 크리에이터 1명당 평가에 **20~30분** 소요 (채널 방문, 영상 검토, 댓글 확인)
- 구독자 수, 조회수, 성장률 등을 **수동으로 계산**해야 함
- 댓글을 하나씩 읽어 시청자 반응을 파악하는 것은 **비효율적**
- 평가자마다 기준이 달라 **일관성이 부족**
- 월 50명 이상의 대규모 스크리닝이 사실상 불가능

### 대상 사용자

- **게임 마케터** (슈퍼센트 파트너십/인플루언서 마케팅 담당자)
- 크리에이터 지원서를 검토하고, 파트너십 확대/유지/축소를 결정하는 의사결정자
- 2차 사용자: **관리자** — 등록된 파트너 풀을 주기적으로 재평가해 계약 갱신 여부를 관리

### 현재 업무 방식

1. 크리에이터가 지원서를 제출 (YouTube 채널 URL 포함)
2. 마케터가 YouTube 채널에 직접 방문
3. 구독자 수 확인 (500명 이상 기준)
4. 최근 업로드 영상 3~5개를 하나씩 클릭하여 조회수/좋아요 확인
5. 댓글을 스크롤하며 시청자 반응 파악
6. 채널의 성장 추세를 감으로 판단
7. 스프레드시트에 수동 기록 후 팀 내 논의

### 주요 불편 사항

| 불편 사항 | 설명 |
|-----------|------|
| 시간 소모 | 1명당 20~30분, 월 50명+ 지원 시 주당 10시간 이상 |
| 댓글 분석 한계 | 수백 개 댓글을 사람이 읽고 감성을 요약하는 것은 사실상 불가능 |
| 기준 비일관 | 담당자별 주관적 판단으로 동일 크리에이터에 대해 다른 결론 도출 가능 |
| 성장률 계산 어려움 | 과거/최근 영상 성과 비교를 위해 별도 정리 필요 |
| 재평가 누락 | 한 번 등록된 크리에이터의 성과 변화를 주기적으로 체크할 수단 부재 |
| 데이터 기반 부족 | 정량 데이터 없이 직감에 의존한 파트너십 의사결정 |

## ② 해결 방법

### 기획하는 AI 제품: **파트너 크리에이터 평가 대시보드 + 관리자 콘솔**

YouTube 채널 URL을 입력하면 **자동으로 채널/영상 데이터를 수집**하고, **AI가 댓글을 감성분석**하며, **종합 점수와 파트너십 추천 결과**를 대시보드로 보여주는 웹 도구. 등록된 크리에이터는 관리자 콘솔에서 **주기적으로 자동 재평가**된다.

### 핵심 기능

1. **채널 자동 분석** — YouTube Data API v3 로 구독자 수·총 조회수·영상 수·가입일·최근 영상·댓글을 한 번에 수집
2. **AI 댓글 감성분석** — Gemini API 로 댓글의 긍정/부정/중립 비율, 핵심 테마, 주목할 댓글(긍2·부2·중1)을 추출. 다량 댓글은 100개씩 배치로 나눠 ThreadPool 로 병렬 호출, 배치별 실제 댓글 수로 가중 평균
3. **5개 항목 가중 종합 점수** — 구독자(20%)·참여도(25%)·성장률(20%)·감성(20%)·일관성(15%) 가중 평균으로 0~100점 계산
4. **PASS / REVIEW / FAIL 자동 판정 + AI 평가 요약문** — 종합 60점 & 구독자 500명을 기준으로 3단 판정. Gemini 가 한국어 요약문을 생성해 마케팅 팀 보고서로 바로 사용 가능
5. **관리자 콘솔** — 등록된 크리에이터 목록을 카드 그리드로 관리, 카드 클릭 시 상세 결과 모달. 스케줄러(cron) 로 전체 평가를 자동 실행 (5 워커 병렬). 외부 노출 금지를 위해 난독화 경로(/sc-9f3b2a7c) + HTTP Basic Auth 로 보호
6. **반응형 대시보드** — PC/모바일 모두 대응. Chart.js 로 4개 차트(조회수·감성·참여율·레이더) 시각화

### 사용자 이용 흐름 (User Flow)

```
[마케터]  개별 분석 흐름
    │
    ├── YouTube 채널 URL 입력
    ├── ('슈퍼센트 게임 영상만 분석' 토글 ON/OFF)
    ├── [분석 시작] 클릭
    │
    ▼
[시스템 파이프라인 (단일 엔드포인트에서 오케스트레이션)]
    ├── ① URL → 채널 ID 해석 (@handle, /channel/UCxxx, /user 등 지원)
    ├── ② YouTube API 로 채널·영상·댓글 수집
    ├── ③ Gemini 로 댓글 배치 병렬 감성분석
    ├── ④ 5개 항목 가중 점수 계산
    ├── ⑤ Gemini 로 한국어 평가 요약문 생성
    └── ⑥ 종합 결과 JSON 반환 (약 15~30초)
    │
    ▼
[마케터]  결과 대시보드 확인
    ├── 추천 카드 (PASS/REVIEW/FAIL 배지 + 점수 + 요약)
    ├── 채널 개요 (썸네일·구독자·조회수·영상수·가입일)
    ├── 차트 2×2 (조회수·감성·참여율·항목 점수 레이더)
    ├── 댓글 감성 패널 (긍중부 + 테마 태그 + 주목 댓글)
    └── 종합 점수 상세 (원형 점수 + 5개 항목 진행바)

[관리자]  재평가 관리 흐름
    │
    ├── /sc-9f3b2a7c (Basic Auth) 로 관리자 콘솔 진입
    ├── 채널 URL 등록 → 풀에 누적
    ├── 스케줄 설정 (매일 09시 등 cron) → 스케줄러가 자동 실행
    ├── 수동 [전체 평가 실행] 버튼 → 5 워커 병렬 처리
    └── 카드 클릭 → 상세 결과 모달 (개별 분석과 동일한 대시보드 재사용)
```

## ⑤ 프로토타이핑 방법

실제 프로토타입은 아래 순서로 **반복적**으로 제작·개선되었다.

### 1단계 — 기본 파이프라인 스캐폴딩

- FastAPI 프로젝트 구조 생성 (backend/ + frontend/)
- 서비스 레이어 4개 분리:
  - `channel_resolver.py` — YouTube URL 다양한 형식을 채널 ID로 변환
  - `youtube_service.py` — YouTube Data API 로 채널·영상·댓글 수집
  - `gemini_service.py` — Gemini 감성분석 + 평가 요약 생성
  - `scoring_service.py` — 5개 항목 가중 점수 계산
- 단일 API 엔드포인트(`/api/v1/analyze/{channel_id}`) 가 전체 파이프라인을 오케스트레이션
- 프론트엔드는 Tailwind CDN + Chart.js + Vanilla JS, 빌드 없이 정적으로 서빙

### 2단계 — 배포 및 영구 저장

- Railway 배포용 `Procfile` 추가
- SQLite 경로를 `DB_PATH` 환경변수로 받아 Railway Volume 에 저장 → 재시작 시 데이터 유실 방지

### 3단계 — 관리자 기능 추가 + 보안 강화

- `manage.html` 을 관리자 콘솔로 구현: 크리에이터 등록·삭제, 전체 평가 실행, cron 스케줄 설정
- 관리자 콘솔을 난독화 경로 `/sc-9f3b2a7c` 로 이동 + HTTP Basic Auth 적용
- 관리용 API 라우터(`/api/v1/creators`, `/api/v1/schedule`) 전부 인증 보호
- 프론트에서 Gemini 반환 댓글 텍스트 escape → XSS 하드닝

### 4단계 — 성능 및 정확도 개선

- 전체 평가 실행을 5 워커 풀로 병렬화
- 한 채널 내부에서도 Gemini 댓글 배치들을 `ThreadPoolExecutor` 로 동시 호출, 워커 수는 `GEMINI_BATCH_CONCURRENCY` 환경변수로 제어
- 긍정/부정/중립 비율을 배치별 실제 댓글 수로 **가중 평균**하도록 수정 (단순 평균 왜곡 해결)
- `evaluate_creator` 엔드포인트를 `async def` → `def` 로 전환해 FastAPI threadpool 에 오프로드 → 평가 중에도 이벤트 루프가 살아있어 사용자가 사이트 탐색 가능

### 5단계 — UX 마감

- 메인 페이지에 "평가 점수 체계" 섹션 추가: 5개 항목 가중치 카드 + 접힘 가능한 상세 구간표 + 판정 기준
- 관리자 카드 그리드를 3열 정사각형으로 정리, 카드 클릭 시 상세 모달
- 모바일 반응형 대응: 헤더·입력·추천·채널·점수 상세를 세로 스택으로, 차트 height 제어, 관리자 카드 `aspect-square` 해제
- 댓글 배치 감성분석 모델을 `gemini-2.0-flash` → `gemini-2.5-flash` 로 교체

### 기술 아키텍처

```
[브라우저]  ──HTTPS──▶  [Railway / FastAPI]
                           ├── /                  (정적 프론트)
                           ├── /api/v1/analyze    (단발 분석)
                           ├── /sc-9f3b2a7c       (관리자 콘솔, BasicAuth)
                           ├── /api/v1/creators   (CRUD, 인증)
                           ├── /api/v1/schedule   (cron 설정, 인증)
                           │
                           ├── YouTube Data API v3
                           ├── Gemini API
                           │   ├── gemini-2.5-flash (배치 감성)
                           │   └── gemini-3-flash-preview (요약)
                           ├── ThreadPool (배치 병렬)
                           ├── Scoring Engine
                           ├── APScheduler (cron)
                           └── SQLite @ $DB_PATH (Railway Volume)
```

## 기능 요구사항
1. YouTube 채널 URL을 입력하면 자동으로 채널 데이터를 수집합니다.
2. YouTube Data API v3로 구독자 수, 조회수, 최근 영상 목록, 댓글을 가져옵니다.
3. Gemini API로 댓글을 감성분석합니다 (긍정/부정/중립, 핵심 테마, 주목할 댓글 5개).
4. 5개 항목으로 종합 점수를 계산합니다:
   - 구독자 점수 (20%): 500명 미만=0, 로그 스케일
   - 참여도 점수 (25%): (좋아요+댓글)/조회수 비율
   - 성장률 점수 (20%): 최근 vs 이전 영상 조회수 비교
   - 감성 점수 (20%): 긍정 댓글 비율
   - 일관성 점수 (15%): 업로드 빈도
5. PASS(60점+, 구독자500+) / REVIEW / FAIL 자동 판정.
6. Gemini가 한국어 평가 요약문을 생성합니다.

## UI 요구사항
- 상단: 타이틀 "파트너 크리에이터 평가 도구"
- URL 입력 필드 + "분석 시작" 버튼 + 로딩 스피너
- 채널 개요 카드: 썸네일, 이름, 구독자, 조회수, 영상 수
- 2x2 차트 그리드: 조회수 추이(바), 감성 분포(도넛), 참여율(라인), 항목별 점수(레이더)
- 댓글 분석 패널: 핵심 테마 태그, 주목할 댓글 목록
- 점수 카드: 항목별 수평 바 + 종합 점수
- 추천 카드: PASS/REVIEW/FAIL 배지(초록/노랑/빨강) + AI 요약문

## 기술 요구사항
- 백엔드: Python FastAPI
- 프론트엔드: HTML + Tailwind CSS CDN + Chart.js CDN + Vanilla JS
- API: YouTube Data API v3 (google-api-python-client), Gemini API (google-genai)
- 환경변수: .env 파일로 YOUTUBE_API_KEY, GEMINI_API_KEY 관리
- FastAPI에서 frontend/ 디렉토리를 정적 파일로 서빙
- 빌드 도구 없이 CDN만 사용

### 6-4. 성능 / 정확도 개선 프롬프트

```
1. [전체 평가 실행] 이 너무 느립니다. 5개 워커 풀로 병렬 처리해주세요.

2. 한 채널의 Gemini 댓글 배치 호출들도 순차 처리되어 느립니다.
   ThreadPoolExecutor 로 동시 호출하도록 바꾸고,
   워커 수는 GEMINI_BATCH_CONCURRENCY 환경변수로 조절하게 해주세요 (기본 3).

3. 긍정/부정/중립 비율을 지금은 배치들의 단순 평균으로 내고 있는데,
   각 배치의 실제 댓글 수로 가중 평균을 내주세요.

4. 전체 평가를 실행 중일 때 다른 페이지가 거의 안 열립니다.
   evaluate_creator 엔드포인트를 async def 에서 그냥 def 로 바꿔서,
   FastAPI 가 threadpool 에 오프로드하도록 해주세요.
   그래야 이벤트 루프가 살아있어서 사용자가 사이트를 돌아다닐 수 있습니다.
```
